# 数独优化功能说明书 (Sudoku Optimization Spec)

## 1. 业务逻辑描述

### 1.1 棋盘渲染性能优化 (Anti-Flicker)
- **目标**：消除玩家交互（点击、填数、切换草稿模式）时的棋盘闪烁感。
- **逻辑**：
  - 渲染器不再盲目执行 `innerHTML = ''`。
  - 采用状态对比机制：仅在单元格类型（数字、草稿、空）发生变化时执行 DOM 重建。
  - 选中状态和高亮状态通过 `classList.toggle` 更新。

### 1.2 草稿同步自动化 (Draft Auto-Clear)
- **目标**：当某一空格填入正确数字后，自动清理相关范围内的草稿。
- **范围**：
  - 同一行 (Row) 的其余 8 个格子。
  - 同一列 (Column) 的其余 8 个格子。
  - 同一 3x3 宫 (Block) 的其余 8 个格子。
- **触发时机**：收到后端的 `sudoku_move_result` 且 `correct: true` 时。

### 1.3 结算弹框最小化 (Dialog Minimization)
- **目标**：允许玩家在游戏结束时暂时隐藏详尽的结算详情，以便截图或查看盘面。
- **交互流程**：
  - 结算弹框右上角增加“最小化”按钮 `[-]`。
  - 点击后，弹框收缩至屏幕边缘（底部或侧边），仅显示简要结果（如“获胜 | 120 vs 80”）。
  - 点击收缩后的条目可重新展开 `[+]`。

## 2. API & 数据定义
- **新方法 (Renderer)**:
  - `updateCellVisuals(row, col)`: 仅更新样式类。
  - `syncDraftsAfterMove(row, col, value)`: 扫描并清理关联格子的草稿。
  - `syncDraftsAfterMove(row, col, value)`: 扫描并清理关联格子的草稿。
  - `toggleMinimization(minimized)`: 切换弹框状态。

### 1.4 结算弹框出现时的位置跳变修复 (Jump Fix)
- **目标**：解决弹框初次渲染时位置不准，瞬间跳向中心的视觉瑕疵。
- **方案**：由于结算弹框使用 Flex 居中，而原有的全局 `popIn` 动画包含 `translate(-50%, -50%)` 导致基准点冲突。现切换为专用的 `sudokuPopIn` 动画，仅保留缩放，确保在 Flex 容器内平稳居中。

## 3. UI 交互流程
1. **游戏结束** -> 显示 Overlay -> 用户点击 `[-]` -> Overlay 执行 CSS 动画收缩，棋盘重新清晰可见。
2. **选中格子** -> 触发 `handleCellClick` -> 调用增量更新 -> 仅目标格及其行列高亮格变更 CSS 类 -> 无视觉闪烁。

## 4. 边缘案例处理
- **草稿清理冲突**：如果用户手动标记了草稿，但随后该数字被确定为该位置的“错解”，草稿不应被清除（当前逻辑下，只有 `correct: true` 才会触发清理，符合预期）。
- **截图覆盖**：最小化后的条目应具有足够透明度或极小面积，不遮挡核心盘面。
